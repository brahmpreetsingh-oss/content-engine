01What Is the Content Engine?

The Content Engine is a Cowork plugin that turns Axelerant's client engagement activity into published marketing content — automatically.

It connects to four data sources (HubSpot CRM, Granola meeting notes, Slack channels, and Google Drive), scans them for "story signals" — moments where something interesting happened on a project — and routes those signals through a series of specialized AI agents that research, classify, write, review, design, and publish the content.

The end output is a fully formatted blog post live on axelerant.com/blog, with a branded feature image, inner diagrams, and a set of distribution assets (LinkedIn posts, newsletter blurbs) ready for the marketing team to deploy.

Key principle: The Content Engine observes what delivery teams are already doing — it doesn't require anyone to "write content." It mines engagement artifacts (meeting notes, Slack threads, CRM notes, project docs) and transforms them into marketing-ready stories.

The most critical agent. Takes a signal card and builds a fully researched, evidence-tagged Story Dossier by reading all surrounding context.

model: opus

Story Classifier

Classifies a dossier into Tier 1/2/3, assigns service scope, engagement theme, content angle, and target persona.

model: inherit

Question Generator

When the dossier has gaps, generates targeted stakeholder questions grouped by role (PM, Tech Lead, etc.).

model: inherit

Narrative Writer

Writes the actual blog post or LinkedIn content from an approved dossier and classification. Follows the blog-writing skill for style and structure.

model: inherit

Content Reviewer

Quality-gates every draft with a type-aware lens — checks technical depth for platform content, sector accuracy for industry content.

model: inherit

Design Agent

Generates Axelerant-branded feature images (800×427px) and inner diagrams using HTML/CSS rendered to PNG. Uploads to HubSpot Files.

model: sonnet

Distribution Agent

Converts approved content into channel-specific assets: LinkedIn posts, newsletter blurbs, email snippets, and distribution sequence plans.

05HubSpot Publishing Workflow

How content gets from a finished draft into a live blog post on axelerant.com.

Architecture: The osascript Bridge

The Cowork VM cannot make outbound HTTPS requests directly (the proxy blocks them). All HubSpot API calls are routed through macOS's osascript tool, which runs curl commands on the host machine where HTTPS is unrestricted.

Publishing Steps

  1. Upload images — Design agent sends feature image and inner diagrams to HubSpot Files API via osascript curl. Each upload returns a public CDN URL (axelerant.com/hubfs/blog-images/...).
  2. Create blog draft — Orchestrator writes the blog HTML payload to a temp JSON file, then POSTs to /cms/v3/blogs/posts. The post is created in DRAFT state.
  3. Attach featured image — PATCH the post with the featuredImage URL and alt text from the image upload step.
  4. Share for review — Post a Slack message in #content-engine-eng with the draft preview link and canvas link, tagging designated reviewers.
  5. Register in publish queue — Add the post to the publish queue JSON with status awaiting-review.
  6. Publish on approval — When a reviewer approves in Slack, the publish monitor calls /push-live to make the post public.

06Slack Review & Approval Process

Every piece of content goes through a two-stage review: AI agent review, then human review via Slack.

Review Workflow

sequenceDiagram participant NW as Narrative Writer participant CR as Content Reviewer participant B as Brahm participant SL as Slack Thread participant REV as Designated Reviewer participant PM as Publish Monitor NW->>CR: Submit draft CR->>CR: Type-aware quality check CR-->>NW: Revisions needed (if any) CR->>B: Draft passed agent review B->>B: Human review + edits B->>SL: Post draft in #content-engine-eng SL->>REV: Tag reviewer + share canvas link Note over SL,REV: 48-hour review window alt Approved REV->>SL: ✅ Approval message PM->>PM: Detect approval in thread PM->>PM: Push live via HubSpot API else Changes Requested REV->>SL: Request changes PM->>B: DM notification B->>NW: Route back for revisions else No Response (48h) PM->>SL: Send reminder ping end
Critical rule: Silence does NOT equal consent. The publish monitor will never auto-publish without an explicit approval message from a designated reviewer in the Slack thread. After 48 hours with no response, it sends a reminder — it does not publish.

What a Review Message Looks Like

When the orchestrator shares a draft for review, the Slack message includes:

  • The blog post title and a one-line summary
  • A link to the full draft canvas (readable content)
  • A link to the HubSpot draft preview (formatted view)
  • Tagged reviewers with explicit ask for approval or feedback
  • A 48-hour review window deadline

Approval Signals

The publish monitor looks for these patterns in the review thread: explicit "approved", "LGTM", "looks good", "go ahead and publish", or a ✅ reaction on the review message. Anything ambiguous is flagged for Brahm rather than auto-publishing.

10Content Tiers Explained

Every story is classified into one of three tiers, which determines the content format, depth, and distribution strategy.

TierFormatWord CountWhen to UseDistribution
Tier 1 Long-form blog 1,500+ words Deep use-case, technical implementation, industry thought leadership with strong evidence Blog + LinkedIn + Newsletter + Email + Campaign
Tier 2 Short blog 600–1,000 words Platform-focused insight, single lesson learned, a notable approach or technique Blog + LinkedIn + Newsletter blurb
Tier 3 LinkedIn post 200–400 words Quick take, milestone celebration, engagement highlight, team shoutout LinkedIn only

Classification Criteria

The Story Classifier evaluates each dossier against these dimensions:

  • Evidence strength — how much verifiable detail exists (facts vs. inferences)
  • External relevance — would someone outside Axelerant care about this story?
  • Technical depth — is there enough implementation detail for a deep dive?
  • Business impact — measurable outcomes or strong implied results?
  • Uniqueness — is this a story only Axelerant can tell?

11Operational Procedures

Day-to-day operations for running the Content Engine.

Weekly Story Mining

Run /mine-engagement once per week (recommended: Monday morning) to scan recent engagement activity across all active projects. The signal fetcher produces Candidate Story Cards which the orchestrator clusters and routes to Story Intelligence for dossier building.

Pipeline State Tracking

The orchestrator maintains a Slack canvas that tracks the current state of every story in the pipeline. This canvas is updated at every stage transition. Check #content-engine-eng for the state canvas link.

Review Workflow Rules

  • Every draft gets a human review before any design or publishing work begins
  • After design and HubSpot draft creation, the post goes to a designated reviewer via Slack
  • The reviewer has 48 hours to approve, request changes, or provide feedback
  • If no response after 48 hours, a reminder is sent — the post is NOT auto-published
  • Only explicit approval triggers publishing

Client Sensitivity

All client names are anonymized throughout the pipeline. The Story Intelligence agent replaces client names with role descriptors ("a US-based healthcare association") and flags any details that could identify the client. Named client use requires explicit written permission noted in the dossier.

Evidence Standards

Every claim in a story dossier is tagged as one of three levels:

TagMeaningRule
[FACT]Directly stated in source materialCan be used as-is in published content
[INFERENCE]Reasonably deduced from evidenceCan be used with hedging language
[INTERPRETATION]Marketing framing added by the agentMust be flagged for human review

Handling Failures

If any stage fails (e.g., HubSpot API error, missing connector, insufficient evidence), the orchestrator halts the pipeline for that story, updates the state canvas with the error, and notifies Brahm via DM. The story can be resumed from the failed stage once the issue is resolved.
